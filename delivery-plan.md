# Smith Delivery Plan

Status: Proposed
Date: 25 June 2026

## 1. Purpose

This plan turns the Smith backlog into an execution sequence. Work is grouped into usable feature slices so each increment can be demonstrated, tested, evidenced, checked in, and managed through the Smith GitHub Project.

The delivery order is deliberately vertical:

1. prove the Code OSS downstream and evidence system;
2. connect to a remote host over SSH;
3. bring up a safe terminal workbench shell;
4. make remote file editing usable;
5. add language and extension value;
6. add daily development workflows;
7. harden for beta and release.

## Product MVP definition

Product MVP means a basic VS Code-like IDE accessible over a terminal through SSH. It is complete only when:

1. `smith <ssh-host> <workspace>` opens a remote workspace over SSH;
2. Smith enters and restores terminal mode safely;
3. the terminal workbench shows navigation, Explorer, editor, panel, and status bar;
4. Explorer browses the remote workspace;
5. the editor opens, edits, undo/redos, saves, and detects conflicts for remote text files;
6. Search finds text in the remote workspace;
7. the integrated terminal runs a remote shell command;
8. the command palette exposes core commands;
9. mouse click, wheel, resize, and keyboard-only operation work for the above;
10. end-to-end evidence is generated against an SSH target.

Foundation slices may be valuable, but they do not by themselves complete the Product MVP.

Requirements remain in [requirements.md](requirements.md), architecture in [design.md](design.md), UX in [ux.md](ux.md), tests in [test.md](test.md), and task details in [backlog.md](backlog.md).

## 2. Delivery rules

### 2.1 Definition of usable

A slice is usable only when:

- a developer can run the documented flow end to end from a clean checkout;
- the flow is connected to real Smith/Code OSS components, not only mocks;
- failures produce actionable messages;
- terminal state is restored after failure or exit;
- the relevant tests pass;
- evidence is written and linked from the issue or pull request;
- the work is committed and pushed.

### 2.2 Definition of done

Every implementation issue must include:

- linked requirement IDs;
- linked test IDs;
- UX section or wireframe where user-visible behavior is involved;
- implementation notes describing reused Code OSS modules and Smith adapters;
- evidence path or artifact reference;
- known limitations and compatibility classification;
- commit or pull request link.

An issue cannot move to `Done` only because code was merged. It moves to `Done` when the feature is demonstrated, tested, evidenced, and documented.

### 2.3 Evidence standard

Each slice produces an evidence bundle under:

```text
test-evidence/<slice-id>-<run-id>/
```

Minimum contents:

- `manifest.json` with repository commit, Smith version, Code OSS commit, client/agent hashes, OS, terminal, SSH topology, test IDs, result, duration, and artifact hashes;
- JUnit or equivalent machine-readable test result;
- logs with default redaction applied;
- terminal frame captures for UI slices;
- compatibility notes for extension or terminal behavior;
- known deviations.

Evidence must not include credentials, private keys, tokens, source secrets, or unrestricted source dumps.

## 3. MVP and staged delivery

### MVP-0: Engineering foundation

**Goal:** A reproducible downstream build, project control model, and evidence harness exist before feature development accelerates.

**User-visible result:** Maintainers can build Smith client and agent artifacts from a pinned Code OSS baseline and see exactly what source, licence, and evidence policy applies.

### MVP-1: Remote connection spine

**Goal:** `smith host /workspace` can resolve SSH configuration, install/start a matching remote agent, authenticate, and open Code OSS remote service channels.

**User-visible result:** A developer can connect to a remote workspace and see connection state, remote host details, and actionable failures.

| Priority | Meaning | Current assignment |
| --- | --- | --- |
| P0 | Required for MVP-0 through MVP-3. Without these Smith cannot be demonstrated as a safe remote terminal editor. | SMITH-001 through SMITH-014, SMITH-016, SMITH-017, SMITH-018, SMITH-019 |
| P1 | Required for MVP-4 and Beta-1. These create high-value IDE workflows after editing is usable. | SMITH-015, SMITH-020 through SMITH-024 |

### 6.2 Labels

Required issue labels:

- `priority:P0`, `priority:P1`, `priority:P2`;
- `phase:MVP-0`, `phase:MVP-1`, `phase:MVP-2`, `phase:MVP-3`, `phase:MVP-4`, `phase:Beta-1`, `phase:Beta-2`;
- `evidence-required`;
- one area label: `area:baseline`, `area:remote`, `area:terminal-ui`, `area:editor`, `area:extensions`, `area:lifecycle`, `area:security`, `area:release`.

### 6.3 Project board policy

The Smith Project board should use:

- `Todo`: accepted work not yet started;
- `In Progress`: implementation active and branch or PR exists;
- `Blocked`: cannot progress without an external decision, credential, upstream fix, or missing scope;
- `Evidence Review`: implementation complete and evidence under review;
- `Done`: merged or committed, pushed, evidence linked, and acceptance criteria met.

If the GitHub Project v2 fields are unavailable to automation, milestones and labels are the source of truth until project-scope access is granted.

## 7. Operating cadence

For each slice:

1. select the next feature bundle from the milestone order;
2. move only the smallest coherent task group into `In Progress`;
3. create or update the implementation branch;
4. implement behind the documented Code OSS/Smith adaptation boundary;
5. run the relevant tests listed in this plan;
6. write evidence under `test-evidence/`;
7. update docs and compatibility notes;
8. commit and push;
9. update issue labels/status/comments with evidence links;
10. move the issue to `Evidence Review`, then `Done` when accepted.

Do not start broad Beta-1/Beta-2 feature work until MVP-3 is demonstrably usable, except for security, licence, evidence, and CI foundations needed to keep earlier work safe.
